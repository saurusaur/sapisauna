# AI 생성 코드 특화 체크리스트

> 바이브코딩 산출물에서만 발생하는 문제 패턴.
> common.md와 함께 항상 로드한다.

## 환각 탐지

> AI는 존재하지 않는 라이브러리를 자신있게 추천한다. 19.7%의 AI 앱에서 발견. (Invicti 2026)

- [ ] import된 패키지가 실제 존재하는지 확인 (npm registry / PyPI / pub.dev 조회)
- [ ] 호출된 API 엔드포인트가 프로젝트 내 정의되어 있는지 확인
- [ ] 참조된 환경 변수가 .env.example 또는 .env에 정의되어 있는지 확인
- [ ] 사용된 라이브러리 메서드/함수가 해당 버전에 실제 존재하는지 확인

## 패턴 일관성 (AI 지질 계층 탐지)

> AI 대화 세션마다 다른 스타일이 쌓인다. 결과: 같은 프로젝트인데 Flask+FastAPI, callbacks+promises+async-await가 공존.

- [ ] 같은 기능에 서로 다른 라이브러리 사용 (예: axios + fetch, moment + dayjs 혼용)
- [ ] 같은 패턴의 다른 구현 (callbacks + promises + async-await 혼재)
- [ ] 같은 데이터에 다른 상태 관리 패턴 혼용 (Context + Redux + Zustand 등)
- [ ] 같은 기능의 다른 스타일 API 라우트 (명명 규칙 불일치)
- [ ] 에러 처리 패턴 불일치 (try-catch, .catch(), error boundary 혼재)

## 코드 중복

> AI 도구 사용 시 코드 중복이 3.1%에서 14.2%로 증가한다. (48% 증가)

- [ ] 유사한 로직의 함수/컴포넌트가 여러 파일에 복제되어 있는지
- [ ] 공통 유틸리티로 추출 가능한 반복 패턴 (3회 이상 반복 시 추출 제안)
- [ ] 거의 동일한 컴포넌트가 약간의 차이로 복제 (props로 통합 가능)

## 테스트 품질

> AI는 "통과"를 최적화하지 "검증"을 최적화하지 않는다. 100% 커버리지인데 버그가 생기는 이유.

- [ ] assertion이 있는지 (빈 테스트 탐지: `test("...", () => {})`)
- [ ] assertion이 실제 동작을 검증하는지 (항상 참: `expect(true).toBe(true)`)
- [ ] mock이 과도하지 않은지 (모든 의존성을 mock하면 실제 동작 테스트 불가)
- [ ] 에지 케이스 테스트 존재 여부 (null, 빈 배열, 경계값)

## 의존성 건전성

> AI가 "그럴듯하지만 존재하지 않는" 패키지를 추천하는 현상(slopsquatting)은 공급망 공격 벡터가 될 수 있다.

- [ ] package.json/requirements.txt/pubspec.yaml의 모든 패키지가 실제 사용되는지
- [ ] 기능이 중복되는 패키지 동시 설치 여부 (lodash + underscore, axios + got 등)
- [ ] 패키지 버전이 고정되어 있는지 (^ 또는 ~ 범위가 과도하게 넓지 않은지)

## 구조 건전성

> 바이브코딩의 근본 문제는 아키텍처 비전 부재. 전문 정리 서비스의 첫 단계가 "아키텍처 감사".

- [ ] 한 파일이 너무 많은 역할을 하지 않는지 (관심사 분리)
- [ ] 순환 의존성 존재 여부
- [ ] 사용되지 않는 파일/모듈 존재 여부
