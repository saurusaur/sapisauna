# PROTOCOL — 먼데이사우나 카톡 크롤링 데이터 추출·검증

> 톡방 CSV 크롤링 데이터에서 사우나 시설/온도/후기 정보를 추출하는 표준 프로토콜.
> 매번 카톡 추출본이 들어올 때마다 이 문서를 따라 진행한다.

작성일: 2026-05-19
최근 적용 추출본: `/Users/stella/Documents/sauna_log/docs/research/katalk-20260519/KakaoTalk_Chat_*.csv`

## 0. 핵심 원칙 (메모리 피드백 반영)

1. **추론 금지** — 시설 분류·도메인 지식은 절대 추측하지 않는다. 패턴 예시를 유저에게 보여주고 의미를 확인받은 뒤 full 분석으로 진행.
2. **스킵 금지** — 오매칭·모호한 항목 발견 시 유저에게 보고하고 해결방안 제안. 임의 결정 없이 결정 포인트만 정리.
3. **컨텍스트 정확도 > 추출량** — 단일 키워드 매칭으로 결론짓지 않는다. 앞뒤 메시지 확인이 사실상 의무.
4. **원문 보존** — 추출 결과에 반드시 원문 텍스트 + 출처(행 번호 + 발화자 + 날짜) 동봉.
5. **에이전트 분할 OK** — 큰 CSV는 청크 분할해 병렬 처리. 단 분할 경계가 컨텍스트 잘림을 만들지 않도록 overlap 유지.

## 2. 추출 Phase 1 — 키워드 매칭 + 컨텍스트 (사우나 후기 본문 외)

### 키워드 셋 (확장 가능)

**사우나 종류**: 건식, 습식, 핀란드, 사우나, 한증막, 불한증막, 불가마, 황토방, 옥돌방, 소금방, 게르마늄

**온도 단서**: `°C`, `도`, `℃`, `90도`, `100도`, 숫자+`도`/`℃`

**시설 키워드**: 노천탕, 외기욕, 외기탕, 냉탕, 급냉탕, 냉수, 얼음, 열탕, 온탕, 식수, 세신, 매점

**지역**: 서울/경기/부산/제주 + 동/구/시 단위 (지역 추천 분류용)

### 추출 룰

- 키워드 매칭된 행: **앞뒤 10건씩 함께 추출** (대화 흐름 컨텍스트)
- 매칭 행 + 앞뒤를 한 단위(unit)로 묶음. 단위 간 중복 메시지는 dedupe.
- 각 unit에 메타 부착: 매칭된 키워드, 행 번호 범위, 시작/종료 날짜, 발화자 수

---

## 3. 추출 Phase 2 — 긴 리뷰 전수 추출

- **100자 이상 단일 메시지는 무조건 통째로 추출** (사우나 후기 가능성 高)
- 그 메시지의 **앞뒤 5건**을 컨텍스트로 함께
- 키워드 매칭과 별개 파이프라인 — 후기 누락 방지

---

## 4. 추출 Phase 3 — 분류 + 검증

### 분류 카테고리 (3분류 우선, 추가 가능)

1. **시설 + 온도 정보** — 특정 사우나 시설의 객관 정보 (예: "○○사우나 핀란드 90도")
2. **지역별 추천** — "○○역 근처 가성비 좋은 곳" 같은 추천 발언
3. **특수 시설** — 노천탕/한증막/외기욕 등 차별화 시설 언급

### 분류 출력 형식 (각 항목)

```markdown
### [장소명 또는 추정명]
- 원문: "<발화 원문 그대로>"
- 출처: row 1234, 2026-04-15 14:23, 사용자 A
- 컨텍스트: row 1230-1238 (앞뒤 메시지 요약 1-2줄)
- 분류: 시설+온도 / 지역추천 / 특수시설
- 추출 정보:
  - 시설명: ...
  - 온도: 건식 90°C / 습식 55°C / ...
  - 지역: ...
  - 특수: 노천탕 ✓ / 한증막 ✓
- 신뢰도: HIGH (명시) / MEDIUM (추정 일부) / LOW (모호)
- 모호 사항: <있으면 명시>
```

### 검증 (유저 팩트체크 포인트)

다음은 추론 금지 항목이며 유저 확인 필요:
- 시설 분류(small-bath / public-bath / hotel-spa / private-sauna / gym-sauna / special)
- 시설 이름 모호 (별명·약어 → 정식명)
- 영업 형태(24h 여부, 가격, 무료 제공 항목)
- 같은 시설로 보이지만 다른 지점 가능성
- "○○ 같다" 류 비교 발언이 사실인지 사용자 의견인지

→ **반드시 "~인지 확인해주세요"로 질문**하며 임의 적용 금지.

## 5. DB 매칭 (Phase 4 — 선택)

- 기존 `places` 테이블과 cross-check
- 새 카톡 추출본에서 발견된 시설 ↔ 기존 DB place 매칭
- 매칭 후보 (HIGH/MEDIUM/LOW 신뢰도)
- 신규 후보 (DB에 없음 → 추가 후보)
- **오매칭 16건 사례** 재발 방지: HIGH 매칭만 자동 처리, MEDIUM/LOW는 유저 검토

산출물: `katalk-db-crosscheck-{YYYYMMDD}.md`

기존 스크립트 활용: `scripts/enrich-places-from-katalk.ts` (시설 보강용)

## 6. 에이전트 분할 전략

### 분할 기준

- **CSV 줄 수 < 3000**: 단일 에이전트로 전체 처리
- **3000-10000**: 청크 2-3개로 분할 (각 ~3000줄, overlap 50줄)
- **10000+**: 청크 5개 이상으로 분할

### 분할 시 컨텍스트 보존

- 청크 경계에 **overlap 50줄** 유지 (긴 후기가 경계를 가로지를 가능성)
- 각 에이전트에 동일 프로토콜 전달
- 결과 병합 시 메시지 행번호 기준 dedupe

### 에이전트 프롬프트 템플릿

```
이 카톡 CSV의 [start-end] 행을 PROTOCOL_katalk_crawl_extract.md에 따라 처리.
- Phase 1: 키워드 + 앞뒤 10건 unit 추출
- Phase 2: 100자 이상 메시지 + 앞뒤 5건
- Phase 3: 3분류로 정리 + 출력 형식 준수
- 모호한 항목은 분류 안 하고 "유저 확인 필요"로 보고
- 산출물: <output_path>에 마크다운으로 저장. 매 항목 원문+출처 동봉.
- 데이터 누락보다 컨텍스트 정확도 우선. 의심 시 컨텍스트 확장 후 보고.
```

## 7. 산출물 명명 규칙 (간소화: 기본 1 + 선택 2)

| 산출물 | 파일명 | 생성 조건 |
|---|---|---|
| **메인 (필수)** | `katalk-extract-{YYYYMMDD}.md` | 항상 — 키워드 unit + 긴 리뷰 + 분류 + 팩트체크 포인트를 섹션으로 통합 |
| DB 매칭 (선택) | `katalk-db-crosscheck-{YYYYMMDD}.md` | DB cross-check 작업 단계 진입할 때만 |
| 구조화 CSV (선택) | `katalk-import-{YYYYMMDD}.csv` | DB import 또는 enrich 스크립트 실행 직전에만 |

날짜는 카톡 추출 시점이 아니라 **분석 실행 일자**.

### 메인 산출물 권장 섹션 구조

```markdown
# 카톡 추출 분석 {YYYYMMDD}

## A. 추출 메타
- 원본 파일, 행수, 청크 분할 계획, 사용 키워드 셋

## B. 키워드 매칭 unit (Phase 1)
- 각 unit: 매칭 키워드 / 행 범위 / 원문 + 컨텍스트 / 메타

## C. 긴 리뷰 (Phase 2, 100자+)
- 각 리뷰: 원문 / 출처 / 앞뒤 5건 요약

## D. 분류 정리 (Phase 3)
- D-1. 시설 + 온도
- D-2. 지역별 추천
- D-3. 특수 시설 (노천탕/한증막/외기욕 등)

## E. 유저 팩트체크 포인트
- 모호한 시설명·분류·영업형태 등 결정 필요 항목 정리
```

Phase별 raw가 분리되지 않아도 섹션 내 원문 인용으로 검증 가능. 진짜 raw가 필요하면 일시적으로 `.raw.md` 임시 파일 만들었다가 메인에 정리 후 삭제.
