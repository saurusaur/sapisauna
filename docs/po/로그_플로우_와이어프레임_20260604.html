0 설계 원칙 — "마찰은 0, 데이터는 남게"

핵심은 레이어(layer)로 분리입니다. 한 화면에 모든 입력을 보여주면 라이트 유저가 도망가고, 입력을 없애면 데이터가 사라집니다. 그래서 ① 발도장 레이어(필수·1탭) → ② 온도 레이어(선택·인라인 1탭) → ③ 상세 레이어(매니아·풀폼) → ④ 저장 후 추가 의 4단을 쌓습니다. 각 레이어는 같은 컬럼/같은 블록을 공유하므로(블록↔슬라이더 동일 데이터) 어느 깊이에서 멈춰도 데이터가 일관됩니다.

레이어누가입력 비용남는 데이터필수?
① 발도장(블록 탭)모든 유저블록 2~4탭 + 어땠어 1탭장소·날짜·루틴 순서·만족도필수
② 온도(인라인)관심 유저블록 위 "+°" 탭 → 슬라이더블록별 온도(건식/냉탕/온탕…)선택
③ 상세(풀폼)매니아"더 자세히" 펼침단계별 시간·반복·청결·비용·세신·메모·동행선택
④ 사후 보강아무나기록 상세에서 편집위 ②③ 무엇이든 나중에선택
우선순위 반영: 유저가 말한 우선순위(온도·루틴·청결도)를 그대로 강제하면 마찰이 생기므로 → 루틴은 ①(필수), 온도는 ②(가장 가까운 선택, 1탭), 청결도는 ③ 상단(상세 펼치면 첫 항목) 으로 배치해 "원하면 즉시, 강제 없이" 채워지게 합니다.

  • 카드 렌더는 블록 시퀀스 1개 소스에서 파생 → 입력 폼·히스토리·스토리가 동일 데이터 사용(이중관리 제거).
  • 기존 renderCard / image-export.ts 재활용. 블록 시퀀스를 받는 레이아웃만 추가.
결정 4 토글을 누적(루틴→온도→시간) 스테퍼로 할지, 독립 칩 3개로 할지. 누적이 단순·직관적, 독립이 표현 자유도 ↑.

6 스키마 스케치 — 단일 세션 테이블 + 블록 자식

유저: "두 테이블 합치는 거 대환영." → logs+deep_logs를 logs 하나로 병합 + 순서/반복/혼합 표현용 log_blocks 신설.

저장 방식 3안

안구조쿼리/집계마이그레이션유연성
1. log_blocks 테이블 추천logs(세션 1행) + log_blocks(N행: seq,type,temp?,dur?,rep?,score?)SQL 집계·필터 자연스러움(장소별 평균온도 등)자식 테이블 추가 + 백필순서·반복·혼합 완전 표현
2. logs.routine JSONBlogs 한 컬럼에 블록 배열집계/필터 어려움(GIN·jsonb_path)컬럼 1개 추가로 끝스키마 자유
3. 하이브리드log_blocks(정) + 자주 쓰는 온도는 logs에 파생 캐시집계 빠름 + 호환트리거/뷰로 동기화둘 다

추천: 1(log_blocks)을 정(正)으로, 3의 파생 캐시를 곁들임. 이유: F3(방문 비교)·장소별 평균온도·통계 인사이트가 전부 SQL 집계 의존이라 JSONB는 부적합. 단, 어드민 시드 온도(238곳)·스토리·통계가 기존 고정 온도 컬럼에 의존하므로, 그 컬럼들을 블록에서 파생되는 캐시로 남겨 무손실·하위호환을 확보.

-- 세션 1행 (logs = 기존 logs + deep_logs 병합) logs( id, user_id, place_id, record_date, visited_at, tribe_id, bath_gender, revisit_score, -- "또 갈래요/어땠어요"(필수) totono_score, water_quality, sweat_quality, rest_quality, cleanliness, -- ③ 점수 cost, currency, companion, crowd, memo, food_eaten, -- ③ deep 병합 primary_sauna_kind, -- 블록서 파생/수동지정 -- ▼ 파생 캐시(블록서 자동 채움, 하위호환·집계·어드민시드용) sauna_temp, steam_sauna_temp, cold_bath_temp, hot_bath_temp, very_hot_bath_temp, ice_bath_temp, jjim_temp, heat_time, ice_time, pause_time, repeat ) -- 블록 시퀀스 N행 (신설, 순서/반복/혼합의 정) log_blocks( id, log_id, seq -- 탭 순서(=루틴), block_type -- 'dry'|'steam'|'cold'|'hot'|'very_hot'|'ice'|'rest'|'scrub'|'store'|'jjim'… (content.ts와 통일), temp -- 선택, duration_sec -- 선택, repeat -- 선택, score -- 블록별 선택 평가, memo ) -- 루틴 템플릿 (나만의 + 추천 시드) user_routines(id, user_id -- NULL=추천시드, tribe_id, name, icon, blocks -- jsonb: [{type,temp?,dur?,rep?}], is_featured)
결정 5 고정 온도 컬럼의 운명: (a) 파생 캐시로 유지(블록이 정, 컬럼은 자동 채움 — 추천, 무손실·호환) vs (b) 완전 제거하고 전부 블록·뷰로 — 깔끔하지만 어드민 시드/스토리/통계 동반수정 범위 큼.
결정 6 어드민 온도 시드(238곳): 현재 logs 고정컬럼에만 존재. 블록 모델로 갈 때 (a) 컬럼 그대로 두고 신규 유저 로그만 블록 병행 vs (b) 시드도 단일 'temp-only' 블록으로 백필. 무손실 우선이면 (a)부터.
결정 7 블록 vocabulary 통일 위치: content.ts의 PLACE_SPECS(HEAT/ICE/PAUSE/BEYOND) id를 블록 타입 SSOT로 쓸지, 별도 BLOCK_TYPES 상수를 둘지. 트라이브 기본 블록 세트 정의도 여기.

7 구현 순서 — 3가지 롤아웃 옵션

옵션 ① UI 먼저 (스키마 시밍)

블록 탭 UI를 먼저 만들고, 저장 시 기존 고정 컬럼으로 매핑하는 shim으로 흘려보냄(순서/반복은 임시로 미저장 또는 JSON 메모). 빠르게 체감 검증.

장점 가장 빠른 출시·검증, DB 변경 0
단점 순서·반복·혼합 데이터가 안 남음 → F3/통계 토대 미완. 나중에 재마이그레이션.

옵션 ② 스키마 먼저

log_blocks·user_routines 마이그레이션 + 병합을 먼저 완료하고 UI를 그 위에 구축.

장점 깨끗한 토대, 재작업 없음
단점 첫 가시 산출물까지 오래. 어드민 시드/스토리/통계 동반수정을 한 번에 떠안음(리스크 집중).

옵션 ③ 증분 병행 추천

1) log_blocks·user_routines를 가산적으로 추가(기존 컬럼은 파생 캐시로 보존) → 2) 블록 탭 UI가 블록 + 캐시 컬럼 둘 다 기록(읽기는 당분간 기존 컬럼 사용 → 스토리/통계 무중단) → 3) 온도 인라인(레이어②) → 4) "더 자세히"(레이어③) + 나만의 루틴 → 5) 읽기를 블록 기반으로 점진 전환, 구 폼 은퇴.

장점
  • 매 단계 배포 가능, 리스크 분산
  • 어드민 시드·스토리·통계 무중단(캐시 컬럼 유지)
  • 실패해도 롤백 쉬움(가산적)
단점 캐시 동기화 코드(블록→컬럼) 한시적 이중관리. 전환 완료 후 제거.

권장 첫 PR: (1) log_blocks/user_routines 마이그레이션 + (2) 블록 탭 빠른기록 화면(레이어①) + 캐시 매핑. 여기까지가 "라이트 마찰 0"의 최소 완성형.
