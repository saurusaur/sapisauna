# 로그 스키마 매핑 · 영향 분석 (병합 전 사전 점검)

작성 2026-06-05 / 브랜치 preview / **목적: 기존 입력·저장·소비 ↔ 신규 블록 모델 간 매칭 깨짐·고아 필드·동반수정 대상 전수 리스트업. DDL 적용 전 단계.**

조사 방법: 실제 코드 대조 (`logs-service.ts`, `sauna-temp-helpers.ts`, `history-stats.ts`, `log/page.tsx`, `log/deep/page.tsx`, `story/page.tsx`, `history/[id]`, `explore/[id]`). ⚠️ 라이브 DB 직접 조회는 미수행 — 컬럼 존재는 service insert/update 기준. 적용 전 `select('*').limit(1)`로 최종 확인 필요.

## 1. 현재 저장 구조 (실측)

### logs 테이블 (세션 1건)
`id, user_id, place_id, record_date, tribe_id, revisit_score, bath_gender, primary_sauna_kind, updated_at`
- **온도(고정 컬럼)**: `sauna_temp`(건식), `steam_sauna_temp`(습식), `cold_bath_temp`(냉탕), `hot_bath_temp`(온탕), `jjim_temp`(한증막)
- **루틴(고정 4슬롯)**: `heat_time`(분), `ice_time`(**초**, migration 016), `pause_time`(분), `repeat`(세트)
- **품질 점수**: `revisit_score`, `totono_score`(사우너), `water_quality`(목욕), `sweat_quality`·`rest_quality`(찜질)

### deep_logs 테이블 (logs 1:1, ON DELETE CASCADE)
`log_id, companion, cost, currency, crowd, memo, cleanliness, food_eaten(배열), has_scrub, scrub_types(배열), scrub_cost, scrub_satisfaction, has_store, store_score, store_memo, has_very_hot_bath, very_hot_bath_temp, has_ice_bath, ice_bath_temp, updated_at`

### ⚠️ 가장 중요한 구조적 사실
1. **온도가 두 테이블로 쪼개짐**: 건식/습식/냉탕/온탕/한증막 → `logs`, **열탕/급냉탕 → `deep_logs`**.
2. **딥로그 저장 시 온도 일부를 logs로 역기록** (`saveOrUpdateDeepLog`): `hot/cold/sauna/steam_sauna_temp`, `primary_sauna_kind`를 **key-presence 협상 로직**으로 처리 — "quick이 소유한 키는 안 건드리고, deep이 보낸 키만 업데이트". 깨지기 쉬운 조정 로직.
3. **온도 입력 → 장소 시설 자동 태깅**: deep 온도/세신 입력 시 `places.facilities`에 `dry-sauna/steam-sauna/very-hot-bath/ice-bath/cold-bath/hot-bath/scrub/massage` 자동 추가 (크라우드소싱).

## 2. 신규 입력 모델 (v4 와이어프레임)

- **블록 시퀀스**: 임의 순서·반복·혼합. 블록마다 `block_type, temp?, duration(분)?, norepeat?`. 전역 `repeat`(세트).
- **완전 퀵**: 블록 + 트라이브 품질 1개(사우너=토토노우/목욕=수질/찜질=땀) + 또갈래요.
- **더 자세히**: 기본=청결도·혼잡도·메모 / 선택=동행·세신·매점·입장료·**기타 시설 온도(루틴 외 시설 일괄)**.
- **탕**: 입력 제거 → `deriveBathGender()` 자동 (현행 quick과 동일, 충돌 없음).

## 4. 매칭 깨지는 지점 (Breakage)

- **B1. 온도 저장 split**: 건식/습식/냉탕/온탕/한증막=logs, 열탕/급냉=deep_logs. 신규는 블록 균일 → "블록 temp를 어느 컬럼에" 라우팅 필요. 또 **같은 타입 블록 2개**(예: 냉탕 2번 다른 온도) → 단일 컬럼으로 표현 불가.
- **B2. ice_time 단위(초)**: 유일하게 초. 신규 타임라인 duration은 분. 변환 + 의미 충돌(냉탕 "1분" vs 기존 "60초").
- **B3. 4고정 슬롯 ↔ 임의 시퀀스**: heat/ice/pause/repeat로는 순서·반복·혼합·norepeat 표현 불가. 역방향(블록→4슬롯) 매핑 시 손실.
- **B4. primary_sauna_kind 의존**: `sauna-temp-helpers.ts`(스토리/히스토리/통계 ΔT)가 이 값에 의존. 블록 모델에서 산출 규칙 재정의 필요.
- **B5. 트라이브 품질 축소**: OLD는 totono/water/sweat **+ rest_quality**(찜질 휴식 퀄) 모두 컬럼. 신규 quick은 트라이브당 1개만 기본 → **rest_quality 입력 경로 소멸**(고아). history detail은 `rest_quality`를 렌더 중(찜질).
- **B6. 신규 온도 시설 = 컬럼 없음**: 불한증막·소금사우나·아이스방·노천탕·아우프구스 등은 저장 컬럼 자체가 없음. → **log_blocks(임의 block_type+temp) 없이는 저장 불가**. (신규에서 추가되는 온도 항목 = 이 지점)
- **B7. 자동 시설 태깅 재구현**: deep 온도 입력→`places.facilities` 자동 태그 로직(`saveOrUpdateDeepLog`)이 고정 컬럼/has_* 기준. 블록/facTemps 기준으로 재작성 필요.
- **B8. key-presence 협상 로직**: quick↔deep 온도 소유권 조정(`setIfPresent`)은 두 폼이 같은 컬럼을 나눠 쓰기 때문. 단일 블록 소스가 되면 **폐기 가능**하나, 전환기 하위호환 주의.
- **B9. 장소 온도 집계(어드민 시드 238)**: `explore/[id]`가 `calcTempAvg('hot/sauna/cold/jjim/steam')` + `calcDeepAvg('very_hot/ice/cleanliness')`로 장소별 평균 온도 표시. 어드민 시드가 logs/deep 고정 컬럼에 있음 → 블록 전환 시 **집계 소스 변경**. 무손실 위해 **컬럼 파생 캐시 유지 권장**.
- **B10. 통계 대시보드**: `history-stats.ts`의 avgRepeat / 주간 heatMinutes(목표 57) / avgDryTempDiff·avgSteamTempDiff(sauna-cold) / avgTotono / avgJjim / avgSweat / avgHotBath / avgWater — 전부 고정 컬럼 집계. 블록 기반 재계산 또는 캐시 필요. `insight-card.tsx` 동반.
- **B11. 스토리/히스토리 렌더**: heat/ice/pause/repeat 뱃지, primary ΔT, deep_log 필드 직접 참조(`story/page.tsx`, `image-export.ts`, `history/[id]`). 블록 시퀀스 렌더로 동반수정.
- **B12. 시설 vocabulary 3중 불일치**: ① `content.ts` PLACE_SPECS(kebab: `dry-sauna, very-hot-bath, cold-bath, bulgama, salt-sauna, open-air-bath`…) ② v4 프로토타입 CAT(short: `dry, very_hot, cold, bulgama, salt, open_air`…) ③ 온도 컬럼명(`sauna_temp=건식, jjim_temp=한증막`). 특히 **한증막**: 컬럼은 `jjim_temp`인데 시설칩은 `bulgama(불한증막)` — 1:1 매칭 모호. **block_type 표준 vocabulary를 SSOT로 확정**해야 함.

## 5. 고아 / 신규 정리

**고아 위험 (OLD에 있으나 신규 UI에 없음)** → 보존할지 폐기할지 결정 필요
- `rest_quality`(찜질 휴식 퀄) — **해소**: 수면/휴식 블록 score(휴식 퀄)로 안착(§10-2). 찜질 quick 기본은 '땀'(sweat) 유지.
- `food_eaten`(deep 배열) — 신규엔 매점 메모만. 폐기 후보.
- `has_*` 불리언 — 신규에선 값 존재로 파생. 컬럼 유지/제거 결정.
- `ice_time` 초 단위 — 분으로 통일 시 기존 값 ÷60 변환.

**신규 (UI에 있으나 OLD 컬럼 없음)** → log_blocks 필요 근거
- 임의 개수·순서(`seq`)·반복(`norepeat`/전역 repeat) 블록
- 같은 타입 다중 블록(냉탕 2회 등)
- 불한증막·소금·아이스방·노천·아우프구스 등 **온도 컬럼 없는 시설**
- 블록별 duration(분, 모든 타입 균일)

## 7. 결론 — 병합 전 확정할 결정

1. **블록↔고정컬럼 관계**: 블록(log_blocks)=정(正), 고정 온도/루틴 컬럼=**파생 캐시 유지**(무손실·어드민시드·집계·스토리 호환). → 권장.
2. **온도 저장 통일**: very_hot/ice도 logs 캐시로 끌어올릴지(현재 deep_logs). 블록이 정이면 캐시 컬럼 위치 자유.
3. **고아 처리**: rest_quality(수면/휴식 블록 score로 부활), food_eaten(폐기), has_*(파생/제거), ice_time(초 통일).
4. **block_type 표준 vocabulary 확정** (content.ts 기준 정리) — B12 해소가 선행.
5. **primary_sauna_kind 산출 규칙**: 첫 heat 블록? 가장 오래/뜨거운? 수동 지정 허용?
6. **마이그레이션 무손실 전략**: 기존 logs/deep → log_blocks 백필(특히 어드민 시드 238) + 캐시 컬럼 병행.

→ 이 6개 확정 후 DDL/구현 (증분 병행안 권장: log_blocks 가산 → 블록+캐시 동시 기록 → 읽기 점진 전환).

## 8. 단일 테이블 병합 목표안 (deep_logs → logs 흡수)

> 전제: **세션 1테이블(`logs`) + 블록 자식(`log_blocks`) + 루틴 템플릿(`user_routines`)**.
> deep_logs는 logs로 흡수(1:1 분리 제거). 순서·반복·신규시설 온도는 1행에 못 담으므로 `log_blocks`는 유지(1:N).

### 8-1. 목표 테이블

```sql
-- logs : 세션 1행 (기존 logs + deep_logs 전부 흡수)
logs(
  id, user_id, place_id, record_date, visited_at, tribe_id, bath_gender,  -- 세션 메타(탕 자동)
  revisit_score,                                                          -- 또갈래요(필수)
  totono_score, water_quality, sweat_quality, rest_quality,              -- 트라이브 품질(quick 1개 + 선택)
  cleanliness, crowd, companion, cost, currency, memo,                   -- ← deep_logs 흡수
  scrub_types[], scrub_cost, scrub_satisfaction,                         -- ← deep_logs 흡수(has_scrub은 파생)
  store_score, store_memo,                                               -- ← deep_logs 흡수(has_store는 파생)
  -- ▼ 블록서 자동 채우는 파생 캐시 (집계·스토리·통계·어드민시드 호환) ▼
  primary_sauna_kind,
  sauna_temp, steam_sauna_temp, cold_bath_temp, hot_bath_temp, jjim_temp,
  very_hot_bath_temp, ice_bath_temp,        -- ← deep_logs서 logs로 끌어올림(온도 저장 통일)
  heat_time, ice_time, pause_time, repeat,  -- 루틴 캐시(블록서 파생)
  created_at, updated_at
)

-- log_blocks : 순서/반복/혼합/신규시설의 정(正)
log_blocks(
  id, log_id→logs(ON DELETE CASCADE), seq,
  block_type, category,  -- 표준 vocab + heat/ice/rest/beyond
  temp, duration_sec,    -- 온도/시간 블록 (초 통일 → ice_time 충돌 해소)
  score, cost, memo,     -- evalOnly 블록(세신/마사지/매점) 평가 — §10-2
  norepeat,              -- 반복 제외(1회)
  created_at
)

-- user_routines : 나만의 + 추천 시드
user_routines(id, user_id NULL=시드, tribe_id, name, icon, blocks jsonb, is_featured)
```

### 8-2. 병합으로 즉시 해소되는 것

| 깨짐 | 단일 테이블 병합 후 |
|---|---|
| B1 온도 split(logs/deep) | ✅ 전 온도 캐시가 `logs` 한 곳. very_hot/ice도 logs로 |
| B8 key-presence 협상 | ✅ **로직 통째 삭제** — quick/deep가 같은 행, 소유권 다툼 없음 |
| deep_logs 1:1 조인 | ✅ 조인 제거 → `toLogWithPlace` 단순화, `LOG_SELECT`에서 `deep_logs(*)` → `log_blocks(*)` |
| `saveOrUpdateDeepLog` 분기 | ✅ 단일 `insertLog/updateLog`가 세션+deep 필드 한 번에 + `insertBlocks()` |

### 8-3. 병합만으론 안 풀려서 log_blocks가 받는 것

- B2 단위충돌 → `duration_sec` 초 통일 (기존 ice_time 그대로, 나머지 ×60 백필).
- B3 임의 순서/반복/혼합/norepeat → log_blocks가 정. logs의 heat/ice/pause/repeat는 **파생 캐시**로 강등.
- B6 신규 온도 시설(불한증막·소금·아이스방·노천…) → log_blocks.block_type+temp (logs 캐시 컬럼은 클래식 7종만).

## 9. 블록 카테고리(heat/ice/rest/beyond) 그룹핑 + log_blocks 필요성 확정

### 9-1. "B6를 근본 해결하면 log_blocks 불필요?" → 아니오
- 필요 근거는 **B3**(임의 순서·반복·혼합 + **블록별 시간**)이지 B6(신규 시설 저장)가 아님.
- 시설별 컬럼을 늘려도(bulgama_temp…) 표현 불가: ① 같은 시설 2번(냉탕→…→냉탕) ② 순서(seq) ③ 블록별 duration ④ norepeat.
- **"각 행동별로 시간을 받는다" = 행마다 시간** → 고정 컬럼은 개수가 안 맞음 → **log_blocks 유지 확정**.

### 9-2. 블록 카테고리 도입 (content.ts를 SSOT)
각 `block_type`에 카테고리 부여(저장 컬럼 `log_blocks.category` 또는 매핑 파생):

| category | 시설 |
|---|---|
| heat(온열) | 건식·습식·온탕·열탕·불한증막·소금·한증막 |

| rest(휴식) | 휴식·외기욕·내기욕 |
| beyond | 세신·마사지·매점·수면·아우프구스 등(시간/온도 의미 약함) |

### 9-3. 카테고리 기반 캐시 파생 규칙 (B3 캐시 강등의 구체화)
- `heat_time` = Σ(heat 블록 duration) — norepeat 아닌 블록은 ×repeat 반영
- `ice_time`(sec) = Σ(ice 블록 duration_sec)
- `pause_time` = Σ(rest 블록 duration)
- `repeat` = 전역 세트
- 온도 캐시(클래식 7종) = 해당 카테고리 내 대표 블록 온도(다중이면 첫/대표 1개)
→ 이 규칙이면 `history-stats`(heatMinutes·avgRepeat 등) **무변경 통과**.

### 9-4. UI/워크플로우 반영 (신규 작업)
- **시설 팔레트(＋추가)**: 현재 HEAT/ICE/PAUSE/BEYOND 그룹 → 카테고리 SSOT와 정합.

### 11-4. 결론
동반수정 = §6(11) + §11-1(신규 6) ≈ **16개 파일**. 단, 표시면은 두 성격으로 구분(앞서 "rename만"은 과소평가):

**(가) 컷오버 필수 — 안 하면 깨짐**: 전 파일 rename + `deep_log.x→x` 평탄화. 특히 **히스토리 상세·record-card·user-log-card·home-calendar·reward-service**는 deep_log 직접 참조라 즉시 깨짐. 장소 상세는 very_hot/ice를 logs 참조로.
**(나) 신규 UX 본체 — 실질 재작성(단계화 가능)**: **스토리 카드**(뱃지→블록 시퀀스+레이어 토글), **히스토리 상세**(→리추얼 타임라인), **장소 상세**(신규 시설 온도+블록 집계). 이게 리디자인의 핵심 산출물.

캐시 전략의 의미 = (가)만으로 **무중단 배포 가능**, (나)는 그 뒤 단계적. 하지만 **결국 스토리/히스토리/장소 셋 다 재작성이 맞음**.

추가 신규 로직: 편집 시 log_blocks 로드/재구성, 건식+습식 메인 사우나 선택, 기타(other) 입력, 세신/매점/수면 평가=타임라인 행, ice_time 라벨 재정의.
