# 스토리 표시 시나리오 설계 — 블록 모델 대응

작성 2026-06-09 / 브랜치 preview / 연계: `handoff_20260607_log_cutover.md` §3 #2(표시면 평탄화)

> **목적**: 입력이 고정 슬롯 → 블록 시퀀스로 바뀌면서, 스토리 카드가 담아야 할 내용도 재설계 필요.
> **불변**: 트라이브별 핵심 메트릭(메인 숫자)은 유지. 그 외(세컨더리)는 조정 가능.

---

## 0. 무엇이 바뀌었나

| | 구 모델 | 새 블록 모델 |
|---|---|---|
| 입력 | 고정 슬롯(사우나/냉탕/온탕/찜질 온도 + heat/ice/pause 시간 + repeat) | 순서 있는 블록 N개. 각 블록 = temp·시간·평가·variant·norepeat |
| 루틴 | 암묵적(시간 슬롯) | **명시적 시퀀스**(seq) + 세트 반복 |
| 범위 | heat/ice/pause만 | **beyond**(세신·마사지·매점·식당·수면·아우프구스) 포함 |
| 표시면 | 고정 메인1 + 뱃지4 + 그래프 | (재설계 대상) |

핵심: **루틴이 다양해졌고**, beyond 활동이 1급 시민이 됨. 현재 고정 4뱃지(HEAT/ICE/PAUSE/RPT)는 실제 한 일을 다 못 담음.

---

## 1. 트라이브 핵심 메트릭 (불변 — 메인 숫자)

| 트라이브 | 라벨 | 값 | 데이터 소스 |
|---|---|---|---|
| saunner | TEMP DELTA | 주 사우나 − 냉탕 (°C차) | `getPrimaryTempDelta` (dry/steam) |
| bather | BATH TEMP | 온탕 온도 (°C) | `hot_bath_temp` |
| jimi | JJIMJIL TEMP | 때면 사우나 최고 (°C) | `getJimiHeadlineTemp` (bulgama/salt/dry/steam) |

---

## 2. 입력 시나리오 축

- **A. 루틴 기록 토글**: ON(온도·시간 기록) / OFF(활동 칩만, 온도 없음)
- **B. 블록 구성**: 기본세트 / heat-only / full-cycle(heat+ice+rest) / beyond 포함
- **C. 반복**: 1세트 / N세트
- **D. 밀도**: 단일 블록 / 다중 블록

## 3. 시나리오 → 표시 매트릭스 (트라이브별 대표 3종)

### 🔵 SAUNNER

| # | 입력 | 채워지는 캐시 | 스토리 hero | 스토리 세컨더리 |
|---|---|---|---|---|
| S1 (이상적) | 건식→냉탕→휴식 ×3, 온도·시간 ON | dry/cold temp, heat/ice/rest time, repeat=3, totono | **TEMP DELTA** = 85−15=70°C | 루틴 시퀀스(건식85·냉15·휴식) + RPT3 + 토토노우 |
| S2 (퀵) | 건식 온도만, 시간X | dry temp만 | **TEMP DELTA** = 70 (냉탕없음→fallback) | 온도만 표시, 시간뱃지 '−' |
| S3 (루틴 OFF) | 활동 칩만(건식·냉탕), 평가만 | temp 없음, totono/revisit만 | hero 숫자 **없음** → 대체 | 평가 도장/활동 아이콘 |
| S4 (변칙) | 한증막+노천만 (건식/습식 X) | bulgama/open_air temp | **현재 '—'** ⚠️ (gap G1) | — |

### 🟢 BATHER

| B1 (이상적) | 온탕→열탕→냉탕, 온도 ON | hot/very_hot/cold temp | **BATH TEMP** = 41 | 탕 시퀀스 + 수질 + 냉탕 |

| B3 (열탕만) | 열탕 온도만 | very_hot temp, hot 없음 | **현재 40 default** ⚠️ (gap G2) | — |

### 🟠 JIMI

| # | 입력 | 채워지는 캐시 | 스토리 hero | 스토리 세컨더리 |
|---|---|---|---|---|
| J1 (이상적) | 한증막→아이스방→수면 | bulgama/ice_room temp, heat/rest time | **JJIMJIL TEMP** = 90 | 시퀀스 + 발한 + 휴식 |

## 4. 발견된 갭 / 결정 포인트

- **G1 (saunner hero 확장)**: `getPrimarySaunaTemp`가 dry/steam만 봄. 한증막/소금/노천 위주 사우너는 hero가 '—'.
  → 사우너 primary에 bulgama/salt/open-air 포함할지? (도메인 판단)
- **G2 (bather hero 폴백)**: BATH TEMP가 hot_bath만. 열탕만 한 경우 default 40 오표시.
  → hot 없으면 very_hot → open_air 순 폴백?
- **G3 (hero 미입력 폴백)**: 루틴 OFF/온도 미입력 시 메인 숫자 없음. 스토리는 공유물이라 hero 공백은 치명적.
  → 대체안: ① 평가 도장(또갈래요/토토노우) 크게 ② 활동 수("4 ACTIVITIES") ③ "기록 완료" 도장
- **G4 (세컨더리 재설계)**: 고정 4뱃지 유지 vs **실제 블록 시퀀스 strip**(아이콘+온도 순서대로) + 반복.
  → 새 모델의 "다양해진 루틴"을 보여주려면 시퀀스 strip이 충실.
- **G5 (beyond 노출)**: 세신/마사지/매점/식당을 스토리에 노출할지. (B2·J2 시나리오)
  → 노출 시: 만족도·가격/맛을 세컨더리 카드로.

---

## 5. 세컨더리 영역 디자인 방향 (택1, 결정 필요)

- **방향 A — 고정 뱃지 유지(현행 확장)**: 메인 + HEAT/ICE/PAUSE/RPT 뱃지 + 그래프. 캐시만 새로 매핑. 구현 최소, 루틴 다양성 못 담음.
- **방향 B — 동적 루틴 strip**: 메인 + 실제 블록 시퀀스(아이콘+온도, 순서대로, 세트 표기) + 핵심 평가. 새 모델 충실, 구현 중간.
- **방향 C — 하이브리드**: 메인 + 루틴 strip + (beyond 있으면) beyond 하이라이트 카드. 가장 표현력 높음, 구현 최대.
