## 날카로운 관찰

**webpack 캐시 크래시를 auth 로직 문제로 오진했다.** 오늘 세션의 가장 큰 시간 낭비. 흰 화면이 뜨자 auth-context를 3번 연속 과잉 수정했는데(getUser 추가 → 백그라운드 검증 → Promise.race 타임아웃), 실제 원인은 `.next` 폴더의 webpack 캐시 오염이었다. 서버 로그에서 JS/CSS 청크 404를 확인한 뒤에야 원인을 찾았다. **증상이 보이면 코드 수정부터 하지 말고, 먼저 서버 로그/네트워크 탭으로 어디서 실패하는지 확인하는 게 맞다.**

**유저 요청 범위를 넘어서는 변경을 자의적으로 했다.** "온열시설 필터 업데이트해줘"라는 요청에 GENDER 필터 라벨과 옵션까지 바꿔서 전부 되돌렸다. HTML 엔티티 `&ldquo;`도 "오타"로 판단하고 유니코드로 바꿨는데 실제로는 정상 작동 중이었다. **변경 범위는 유저가 명시한 것만. "이것도 고치면 좋겠다"는 별도로 제안하고 승인 받은 뒤에.**

**middleware.ts가 4회 수정됐다.** 이번 주 가장 많이 수정된 파일 중 하나(BACKLOG 6회 제외). auth 관련 변경이 반복되면서 middleware까지 연쇄적으로 건드려진 패턴. auth 레이어 변경 시 영향 범위를 미리 확인하는 습관이 필요하다.

**P0 완료율은 높지만 P1 이월이 쌓이고 있다.** P0 4건 완료(장소 DB 연동, places-service 업데이트, 타입 동기화, 지도 URL). P1은 5건 완료했지만 8건이 여전히 열려 있다. 특히 "중복 로직 제거"는 3/1부터 열려 있는데 아직 손을 못 댔다. 기능 추가에 밀려서 기술 부채 정리가 계속 후순위로 밀리는 전형적 패턴.

## 잘 된 것

**DB 전환을 단계적으로 완료했다.** places → logs → 레거시 삭제 순서로 진행한 건 올바른 전략이었다. 한 번에 몰아서 하지 않고, 각 단계마다 커밋을 찍으면서 롤백 가능한 상태를 유지했다.

**currentLog 버그를 체계적으로 추적했다.** "새 로그에 이전 데이터가 채워진다"는 버그를 3개 진입 경로(place/page, place/add, explore/[id])를 모두 찾아서 일괄 수정. 하나만 고치고 끝내지 않은 건 좋은 습관.

**앱 안정화 4건을 최소 변경으로 마감했다.** auth-context 과잉 수정 삽질 후 결국 원래 계획(.catch 1줄)으로 돌아왔다. 최종 결과물은 깔끔하다: .catch 1줄, try/catch 2곳, error.tsx 1개.

## 다음 주에 달라져야 할 것

1. **"흰 화면" → 코드 수정 전에 네트워크/서버 로그부터 확인.** webpack 캐시, 빌드 실패, 404 같은 인프라 문제를 코드 레벨 버그로 오진하면 시간을 3배 쓴다. `rm -rf .next && npm run dev`를 먼저 시도하는 체크리스트를 만들자.

2. **변경 범위를 유저 요청에 한정.** "이것도 같이 고치면 좋겠다"는 제안으로만 언급하고, 승인 없이 코드에 반영하지 않는다. 오늘 GENDER 필터 되돌린 시간이 아까웠다.

3. **P1 리팩토링에 시간을 할당.** 중복 로직 12건이 3일째 열려 있다. 다음 세션에서 기능 작업 전에 최소 1건(예: getFacilityLabel 통합)을 먼저 처리하자.

# 디버그 우선 규칙

## 흰 화면 / 페이지 미로딩 시 체크리스트
1. 브라우저 콘솔 에러 확인

## 원칙
- 증상을 보고 코드부터 수정하지 않는다
- 서버 로그 + 네트워크 탭으로 실패 지점을 먼저 특정한다
- 인프라 문제(캐시, 빌드)와 코드 문제를 구분한 뒤에 수정을 시작한다
