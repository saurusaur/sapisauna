## 날카로운 관찰

**진단 정확도 문제가 있었다.** 화이트 스크린을 처음에 "UserProvider가 null 반환"으로 진단하고 15파일 방어 코드 기획을 세웠는데, 반박(devils-advocate) 돌리고 나서야 진짜 원인이 placeholder Supabase 키라는 걸 잡았다. 기획을 실행했으면 불필요한 코드 15파일에 뿌렸을 것이다. **반박 프로세스가 과잉 구현을 막아준 첫 번째 실전 사례.**

**"코드 버그"와 "환경 설정"을 구분하는 데 시간이 걸렸다.** SSL 인증서 문제도 `npm run dev` vs `npx next dev` 차이를 놓쳐서 한 번 더 삽질했다. 사우너 그래프 버그도 에이전트가 긴 분석을 했지만 결국 사용자 오인이었다. 코드를 의심하기 전에 환경/데이터부터 확인하는 습관이 필요하다.

**테스트 없이 제안한 적이 있었다.** 네이버 "사우나" 강제 추가를 제거하자고 했다가 사용자가 "테스트 해보고 말하는 거야?"라고 지적했다. 실제로 테스트해보니 "해운대" 검색 시 사우나 추가가 있으면 목욕탕이 나오고, 없으면 홈플러스가 나왔다. 데이터 없는 제안은 위험하다.

## 다음에 달라져야 할 것

**1. "코드 전에 환경" 원칙.** 버그 리포트를 받으면 먼저 (1) env vars (2) 네트워크/인증서 (3) 브라우저 콘솔을 확인하고, 그 다음에 코드를 의심한다. 오늘 화이트 스크린, SSL, 사우너 그래프 세 건 모두 코드 문제가 아니었다.

**2. 제안 전 반드시 실측.** "~하는 게 나을 것 같다"라고 말하기 전에 최소 하나의 테스트 결과를 붙인다. 네이버 검색 필터 건에서 배운 교훈.

**3. `npx` vs `npm run` 차이 주의.** package.json scripts에 환경변수가 설정되어 있으면 반드시 `npm run dev`로 실행. `npx next dev`는 스크립트를 우회한다.
