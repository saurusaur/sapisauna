## 1. 배경 / 현황
- `/admin` 라우트 **없음**(신규 구축 대상).
- 현재 어드민 동작 방식:
  - **장소 등록·교정** = `scripts/katalk-*` 스크립트로만(비개발 운영 불가).
  - **큐레이션(is_featured)** = `toggle_featured` RPC(마이그레이션 012), sa-list 상세에서 어드민만 토글. 어드민 식별 = **`ADMIN_USER_ID` 하드코딩**(`src/constants/content.ts`).
  - **어드민 온도 로그** = `ADMIN_USER_ID`로 작성, 유저 노출 제외(`logs-service.ts`).
- 문제: 큐레이션·등록·교정·신고처리용 **운영 UI 부재** → P0 큐레이션 시드·F6 운영의 병목.

## 2. 권한 모델 (⚠️ 결정 필요)
- 현재: 단일 `ADMIN_USER_ID` 하드코딩(1인).
- 옵션: **(a) 하드코딩 ID 유지**(빠름, 솔로 운영) / (b) `profiles.role` 컬럼 추가(확장·다인) / (c) env allowlist.
- 게이트: `middleware.ts`에서 `/admin` 보호(현재 public: /login·/explore·/api) + **서버에서 재검증**.

## 3. 기능 그룹 (모듈 그룹핑 — 어드민 사이드바 섹션)
11개 평면 모듈 → **7개 그룹(어드민 사이드바 섹션)**으로 그룹핑. (NEW=이번 추가)

### A. 운영 대시보드 & 모니터링
- KPI: 유저·로그·장소 수, 트라이브 분포, 신규 가입·활동 추이
- 데이터 품질 리포트: `katalk-db-full-audit.mjs` 플래그·온도 이상치·resort-spa 재분류 후보
- **(NEW) 시설 완성도 리포트** (`places.facilities` 집계): ①완성도 분포(빈 곳 %·평균 개수·5/8/10개+ 비율 — *현재 312곳 기준 빈 4%·평균8.5·10개+41%*) ②**가장 많이 쓰는 시설 top**(현재 tattoo-friendly85%·cold-bath78%·hot-bath76%·dry-sauna76%…) ③**유저 요청 시설 집계**(아래 유저 기여 데이터 필요) ④시설 빈/희소 장소 목록 → B 교정·유저 기여로 연결
- **(NEW) 유저 시설 기여/요청 플로우** (크로스): 장소 상세에 "시설 추가/수정 제안" → D 검수 큐로 유입 → 반영 시 recognition(이중가치 루프). ※백로그 P3 "시설 태그 유저 기여" 일반화. 이게 있어야 ③ "요청 시설" 집계 가능
- (NEW) 시스템/에러 모니터링(Sentry 요약 링크) · **어드민 감사 로그**(누가 무엇을 변경)

### B. 장소·데이터 관리
- 장소: 검색/목록, **수동 등록**, 편집(facility_type·bath_policy·city·facilities·24h), 병합, 폐업
- 데이터 교정: place_sources(이름·주소·source·external_id)·좌표·**어드민 온도 로그**
- ※온도 교정 화면은 **L1 로그·스키마 결과**에 영향받음

### C. 큐레이션 운영
- **(NEW) 어드민 직접 리스트 생성·편집** — 테마 큐레이션 리스트를 어드민이 직접 생성(장소 추가·커버·이모지·설명·태그) → 어드민 소유 featured 리스트. 기존 `list-form`·`create-list-sheet` 로직 재사용
- is_featured SA-리스트 관리: 유저 리스트 추천 등록/해제·노출 순서
- FEATURED 콜드공백 선충전(전국 노천탕 Top 등 시드 리스트)

### D. 인박스·검수 (인바운드 처리)
- 검수 큐: 유저 신고("다른 장소"/"폐업"), 수동 등록 리뷰

### E. 유저·계정 관리 (NEW)

## 4. MVP 우선순위 (그룹 기준)
- **P0 (베타 운영 필수)**: **B** 장소 등록/편집/교정 + **C** 큐레이션 → 시드·직접 운영.
- **P1**: **D** 인박스·검수(+업체등록·피드백 폼) · **E** 유저 활동상태·스테이터스(문제 유저 대응) · **A** 대시보드.
- **P2**: **F** 문구·용어·컬러·로고(CMS 결정 후) · A 감사로그/에러 모니터링 · B 병합/폐업 자동화.
- **P3**: **G** 공지·알림 · F 디자인 에셋 풀 CMS.

## 5. 기술 메모
- 라우트 `/admin/*`(App Router) + 미들웨어 가드 + 서버 재검증.
- 쓰기 다수가 **service_role 필요** → **서버 route handler**로 처리(클라에서 service key 직접 사용 금지).
- 기존 스크립트 로직(**등록 컨벤션·온도 범위검증·좌표 프록시미티 중복가드**, `katalk-new-register`/`katalk-overseas-register`)을 서버 API로 이식·재사용.
- 데이터 모델은 **L1 로그·스키마 재검토 결과에 영향받음**(B 온도 교정) → B 온도·A 품질리포트는 L1 이후 확정.
- **에셋/로고(F)**: Supabase Storage 버킷 + 메타 테이블, 런타임 참조. PWA manifest·favicon은 빌드 자산이라 런타임 교체 가능 범위 확인 필요.
- **콘텐츠/컬러(F)**: 코드 상수 유지 시 어드민은 "편집 제안→PR/배포" 흐름, DB 이전 시 런타임 fetch + 캐시. 이중관리·캐시 무효화 트레이드오프.
- **유저 상태(E)**: `profiles`에 status 컬럼(active/warn/suspended/withdrawn) 신설 + RLS 영향 검토. 활동 집계는 logs/place_sources 조인.

## 6. 미결정 (유저 확인 필요)
1. **권한 방식**: 하드코딩 ID 유지 vs `profiles.role` 신설(향후 운영자 추가 계획?).
2. **등록 UI 범위**: 대량=스크립트 유지 / 단건·교정=UI? 어디까지 UI화?
3. **큐레이션 소유 모델**: 어드민 직접 리스트 생성(C)을 넣기로 → **어드민 소유 리스트** 방식 채택 방향. 확인할 것: 전용 어드민 계정(`ADMIN_USER_ID`) 소유로 둘지 / RLS상 어드민 생성·편집 허용 경로(현재 `toggle_featured` RPC처럼 owner-only 우회 필요).
4. **검수 큐 신고 입력 경로**: 유저 신고 UI(장소 상세 "신고") 선행 필요 — 별도 백로그 연계.
5. **콘텐츠·컬러·로고(F) 관리 아키텍처 (핵심)**: 현재 코드 상수(`content.ts`·`story-colors.ts`·빌드 자산) → **(a) 코드 유지**(어드민=편집 가이드/제안, 반영은 배포) vs **(b) DB CMS화**(런타임 fetch+캐시, 즉시 반영). 범위·우선순위 결정 필요.
6. **인박스(D)**: 수신 채널 = 앱 내 폼만 vs 이메일(sapi.sauna@gmail.com) 연동 vs 둘 다. 카테고리(피드백/제휴/**업체 등록**)별 폼·처리 흐름 분리 범위.
7. **유저 관리(E)**: status 값 체계(정상/주의/정지/탈퇴)와 각 상태의 앱 동작(정지 시 로그인·기록 차단 등) 정의. 회원 탈퇴를 이메일 요청 → 어드민 처리로 전환할지.

## 8. 검토 — 비어있는 부분·과한 부분 (2026-06-06)

> 이 문서는 6/4 작성. 그 뒤 확정된 전제 3가지를 반영해 재검토. 결론: **빠진 건 2티어 데이터 운영, 과한 건 모더레이션·CMS.**

### 8.1 바뀐 전제 (재검토 기준)
- **2티어 5천 공공데이터 확정** — 영업중 ~5,000곳(기본티어) + 검수 312곳(리치티어) 전량 노출. → B·A에 직접 영향.
- **애널리틱스 = 어드민 대시보드(A)와 함께 개발자 친구 구현 확정.**
- **베타 = 인플루언서 초대형(~150) 클로즈드**(유입은 유저가 인스타로 직접). → 어뷰즈·모더레이션 리스크 낮음, 앱내 게이팅 불필요.

### 8.2 🔴 비어있는 부분 (Gaps)
1. **2티어(기본 5천) 운영 도구** — 단건 편집만으론 5천 못 다룸. 필요: **일괄**(영업상태 갱신·중복 정리)·**티어 승격**(기본→리치)·**기본핀 노출 토글**·좌표 이상/누락 탐지. → **B의 가장 큰 공백.**
2. **품질 리포트 5천 확장** — A 품질리포트가 312 기준(katalk audit). 5천용으로 좌표결측·중복의심(312↔5천 / 5천 내부)·티어 분포 추가.
3. **유저 장소 신고·교정 동선** — 미검증 5천이 들어오면 "폐업했어요/정보 틀려요" 신고가 품질 유지의 핵심 피드백. §6.4가 "신고 UI 선행 필요"로만 남겨둠 → **5천 때문에 우선순위 ↑**, 베타에 가벼운 신고버튼+큐는 살려야 함.
4. **제품 퍼널 애널리틱스** — A가 내부 KPI+Sentry만. 유저 행동 퍼널(가입→온보딩→첫기록→재방문) 명시 추가(개발자 담당 확정).
5. **베타테스터 활동 한눈에** — E "베타테스터 관리" 한 줄을 구체화(활성/이탈·첫기록 전환). A 애널리틱스와 겹침 → **통합**.

### 8.3 🟡 과한 부분 (Overkill — 베타·솔로 기준 빼거나 미룸)
1. **F 브랜드·CMS 전체** — 솔로는 코드상수+배포로 충분. **GLOSSARY만 코드상수**(뉴비 용어팝업용)로 두고 로고/컬러/약관 CMS화는 post-beta. → F는 베타 어드민에서 제외.
2. **E 유저 status/차단/정지 머신 + RLS** — 초대형 클로즈드 베타는 어뷰즈 ~0. status 컬럼·RLS·차단은 과함 → 베타선 **read-only 모니터링만**, 탈퇴는 이메일 유지. post-beta.
3. **D 문의 인박스** — **복잡도는 "방향"에서 갈림**(유저 질문에 답):
   - ✅ **인바운드 폼 + 어드민 큐(읽음·상태·메모)** = **안 복잡함**. 테이블 1개(`inquiries`: user_id·category·message·status) + 제출 폼 1개 + 어드민 리스트 1개. → **만들 만함**. 카테고리(피드백/제휴/업체등록)별로 정리돼 메일 흩어짐보다 나음.
   - ❌ **양방향 메시징(유저가 답장 보고·알림)** = **복잡함**(스레드·읽음·알림). 알림은 푸시(V1.5)까지 끌고 옴 → **베타 스킵**.
   - 🟰 **권장 = 가벼운 인바운드 폼+큐 + 답변은 메일(sapi.sauna)**. 정 부담되면 전부 메일도 OK(베타 볼륨 작음).
   - (§8.2-3 **장소 신고**는 이 폼의 한 카테고리로 흡수 가능 — 데이터 품질 직결이라 살림.)
4. **A 감사 로그** — 어드민 1인이라 "누가 바꿈"이 무의미. multi-admin 전까지 불필요.
5. **A 유저 시설 기여/요청 플로우** — 유저 대면 기능(제안 UI+큐+테이블)이 어드민에 끼어있음. 별 기능·P3 → 베타 제외/디커플.
6. **G 공지·알림(푸시)** — V1.5 → 베타 제외 확정.
