## 0. 스토리 카드 → 기록 상세 레퍼런싱

### 스토리 카드의 정보 구조 (타입별)

```
사우너파 스토리:
┌─────────────────────┐
│ 장소명               │
│ 날짜 · 요일          │
│                     │
│ TEMP DELTA          │ ← 라벨
│ 65°C               │ ← 메인 숫자 (사우나-냉탕 차이)
│                     │
│ 10  2   5   3      │ ← HEAT ICE PAUSE RPT (루틴)
│                     │
│ [사우너 그래프]      │ ← 온도+세트 시각화
│                     │
│ ● SAUNNER           │
└─────────────────────┘

목욕파 스토리:
│ BATH TEMP           │
│ 42°C               │ ← 목욕물 온도
│ [수질+온도 그래프]   │

찜질파 스토리:
│ JJIMJIL TEMP        │
│ 75°C               │ ← 한증막 온도
│ [휴식퀄+온도 그래프] │
```

### 스토리에서 가져올 디자인 언어

| 스토리 요소 | 기록 상세 적용 |
|-----------|--------------|
| 타입별 라벨 (TEMP DELTA, BATH TEMP...) | 메트릭 영역 상단 영문 라벨 |
| 대형 숫자 + 작은 단위 | 온도 대시보드 동일 패턴 |
| HEAT/ICE/PAUSE/RPT 4컬럼 | 루틴 표시 (옵션 R2와 직접 대응) |
| 타입 dot + 영문명 | 장소 카드 내 타입 표시 |
| 타입별 컬러 시스템 | 메트릭 영역 악센트에 활용 가능 |

### 핵심 인사이트: "기록 상세 = 스토리 카드의 확장판"

스토리 카드가 **핵심 1개 숫자**에 집중한다면,
기록 상세는 **모든 숫자를 펼쳐서** 보여주는 역할.

→ 스토리의 시각 언어(영문 라벨, 대형 숫자, 루틴 4컬럼)를
   기록 상세에서도 사용하면 앱 내 일관성 극대화.

## 1. 로그 폼 ↔ 기록 상세 디자인 레퍼런싱 포인트

### 퀵로그 폼 (입력)에서 가져올 수 있는 패턴

| 퀵로그 폼 요소 | 상세 페이지 대응 | 레퍼런싱 포인트 |
|--------------|----------------|---------------|
| 장소명 h1 볼드 24px | 장소 카드 내 장소명 | **동일 폰트 사이즈/웨이트** 사용 |
| 타입 이모지+드롭다운 | 타입 표시 | 이모지를 장소 카드 우측에 배치 (Lovable처럼) |
| 슬라이더 + 대형 숫자 | 대형 메트릭 숫자 | **폼에서 표시하는 숫자 스타일을 상세에서도 동일하게** |
| 넘버 칩 (토토노이/재방문) | 점수 라벨 텍스트 | 폼=칩 선택, 상세=결과 라벨 (입력↔출력 대응) |
| ROUTINE glass-card-light | 루틴 표시 영역 | **동일 glass-card-light 배경** |
| 플로팅 CTA 버튼 스타일 | 스토리 만들기 CTA | **동일 그림자 + rounded-2xl** |

### 딥로그 폼 (입력)에서 가져올 수 있는 패턴

| 딥로그 폼 요소 | 상세 페이지 대응 | 레퍼런싱 포인트 |
|--------------|----------------|---------------|
| 섹션별 제목 (볼드, 한글) | Deep Log 헤더 | 폼="동행자", 상세="동행자" → **동일 라벨 스타일** |
| 칩 선택 UI | 선택된 값 텍스트 | 입력=칩 그리드, 출력=텍스트 (자연스러운 대응) |
| glass-input 텍스트 필드 | 메모 표시 | **glass 배경 통일** |
| border-t border-stone-200/60 구분 | 항목 간 구분 | 폼과 동일한 얇은 구분선 or 여백 |

### 핵심 원칙: "입력 폼의 읽기 전용 버전"
- 폼에서 슬라이더로 조작한 숫자 → 상세에서 대형으로 표시
- 폼에서 칩으로 선택한 값 → 상세에서 텍스트로 표시
- 폼의 glass 스타일 → 상세에서도 동일한 glass 스타일

## 2. 전 타입 대응 레이아웃 검증

### 타입별 메트릭 매핑

| | 사우너파 🔥 | 목욕파 🛁 | 찜질파 🥚 |
|---|-----------|---------|---------|
| **메트릭 1** | 사우나 온도 (°C) | 목욕물 온도 (°C) | 한증막 온도 (°C) |
| **메트릭 2** | 냉탕 온도 (°C) | 수질 (1-5 라벨) | 휴식 퀄리티 (1-5 라벨) |
| **메트릭 3** | — | 냉탕 온도 (°C, 선택) | — |
| **점수 1** | 토토노이 (1-5 라벨) | — | — |
| **점수 2** | 재방문 의향 (1-5 라벨) | 재방문 의향 | 재방문 의향 |
| **루틴** | HEAT/ICE/PAUSE/REPEAT (필수) | HEAT/ICE/PAUSE/REPEAT (선택) | HEAT/ICE/PAUSE/REPEAT (선택) |

### ⚠️ Lovable 3컬럼 레이아웃의 문제

Lovable 시안은 사우너파 전용 (85° / 15° / 3 SETS = 3컬럼):
- **찜질파**: 온도 1개만 → 3컬럼 중 2칸 비어 불균형
- **목욕파**: 수질이 "생명수" 같은 텍스트 → 숫자가 아니라 대형 표시 어색
- **목욕파 냉탕 선택 시**: 3개지만, 수질은 온도와 성격 다름

### ✅ 해결: 유연한 그리드

```
온도류 = 대형 숫자 (°C 단위가 있는 것)
점수류 = 라벨 텍스트 (토토노이, 수질, 휴식 퀄리티, 재방문)
```

**온도 영역**: 있는 만큼만 균등 분할 (1개=전체폭, 2개=2컬럼, 3개=3컬럼)
**점수 영역**: key-value 로우 (현재 Lovable의 토토노이/재방문 스타일)

타입별 렌더링:

```
🔥 사우너파:
┌─────────┬─────────┐
│  85°C   │  15°C   │  ← 2컬럼 (온도 2개)
│  HEAT   │  ICE    │
└─────────┴─────────┘
  토토노이 강도     좋음     ← 점수 로우
  재방문 의향   근처면 갈만   ← 점수 로우

🛁 목욕파:
┌─────────┬─────────┐
│  42°C   │  15°C   │  ← 2컬럼 (온탕 + 냉탕 선택시)
│  HOT    │  ICE    │
└─────────┴─────────┘
  수질            청량     ← 점수 로우
  재방문 의향      최고     ← 점수 로우

🛁 목욕파 (냉탕 미선택):
┌───────────────────┐
│      42°C         │  ← 1컬럼
│      HOT          │
└───────────────────┘
  수질            청량
  재방문 의향      최고

🥚 찜질파:
┌───────────────────┐
│      75°C         │  ← 1컬럼
│      HEAT         │
└───────────────────┘
  휴식 퀄리티      편안     ← 점수 로우
  재방문 의향      인생     ← 점수 로우
```

## 3. 딥로그 구분선 — 필요한가?

### Lovable: border-b 구분선 있음
```
┌─────────────────────────┐
│ 탕 선택          남탕    │
│─────────────────────────│ ← border-b
│ 동행자           친구    │
│─────────────────────────│
│ 방문 목적        운동 후  │
│─────────────────────────│
│ ...                     │
└─────────────────────────┘
```

### 분석: **불필요**
- 딥로그 항목은 3-7개 (유동적 — 세신/매점/메모는 있을 때만)
- 현재 앱의 key-value 로우에 이미 `py-3` 여백이 있어 시각적 분리 충분
- 딥로그 폼(`log/deep`)에서도 **구분선 없이 여백만으로 분리** 중
- 구분선 넣으면 항목 적을 때 (3개) 과하게 느껴짐

### 제안: **구분선 제거, 여백으로 분리**
- `space-y-3` 또는 `py-3`으로 충분
- 폼과 동일한 "여백 기반 분리" 패턴 유지
- 깔끔하고 미니멀

```

```
┌─ DEEP LOG ─────────────────┐
│ 동행자           친구       │
│ 방문 목적        운동 후    │
│ ...                        │
│                            │
│ 💬 운동 후 갔는데 역시...   │  ← 메모를 카드 하단에
└────────────────────────────┘
```

**옵션 B: 별도 glass tile (추천)**
```
┌─ DEEP LOG ─────────────────┐
│ 동행자           친구       │
│ 방문 목적        운동 후    │
│ ...                        │
└────────────────────────────┘

┌────────────────────────────┐
│ 운동 후 갔는데 역시 좋다.   │  ← glass-card-light

## 5. 키 컬러 사용 전략

### 원칙: "평가에만 레드, 정보는 뉴트럴"

| 요소 | 성격 | 컬러 |
|------|------|------|
| 온도 숫자 (85°C, 15°C) | 정보/측정값 | **text-stone-800** (뉴트럴 다크) |
| °C 단위 | 보조 | text-stone-400 |
| 라벨 (HEAT, ICE) | 보조 | text-stone-400 |
| 토토노이/수질/휴식 값 | 주관적 평가 | **var(--color-accent) 레드** |
| 재방문 의향 값 | 핵심 평가 | **var(--color-accent) 레드 + 볼드** |
| 루틴 숫자 | 정보/시간 | text-stone-700 |
| 딥로그 값 | 정보 | text-stone-700 font-bold |
| 매점 만족도 | 평가 | **var(--color-accent) 레드** |

Lovable에서 HEAT=레드, ICE=틸로 온도에 컬러를 줬지만:
- 3가지 타입에서 온도 의미가 다름 (사우나 온도, 목욕물 온도, 한증막 온도)
- 타입 컬러와 혼동 (파/주/초 vs 빨/초)
- **정보에는 뉴트럴** 원칙이 더 일관적

## 6. 루틴 표시 — 옵션 비교

### 현재 상태
기록 상세에 루틴 미표시 (세트 수만 메트릭으로 표시)

### Lovable 시안
3컬럼 글래스 칩: `10 MINS HEAT | 2 MINS ICE | 5 MINS REST`
→ 별도 글래스 패널, 시각적으로 무거움

### 요청: "좀 더 모던하고 깔끔하고 가독성 좋게"

---

#### 옵션 R1: 인라인 시퀀스 (추천)

```
🔥 10분 → 🧊 2분 → 😮‍💨 5분  × 3세트
```

- 한 줄로 루틴 전체가 읽힘
- 화살표(→)가 순서/흐름을 직관적으로 표현
- 가장 컴팩트, 추가 카드 불필요
- 폼의 ROUTINE 섹션과 대응하면서도 "결과 요약"느낌
- 메트릭 영역 바로 아래, 같은 카드 안에 배치 가능

```
┌─────────┬─────────┐
│  85°C   │  15°C   │
│  HEAT   │  ICE    │
├─────────┴─────────┤
│ 🔥10분 → 🧊2분 → 😮‍💨5분 × 3 │  ← 작은 텍스트, stone-500
└───────────────────┘
```

**장점**: 미니멀, 시퀀스가 한눈에, 카드 추가 없음
**단점**: 값이 없는 항목 처리 필요 (ICE=0이면 생략)

---

#### 옵션 R2: 타이포 매트릭스

```
┌───────────────────────────┐
│  HEAT   ICE   PAUSE   SET │  ← stone-400 라벨
│  10분   2분    5분     3  │  ← stone-700 볼드 값
└───────────────────────────┘
```

- Lovable의 3컬럼을 4컬럼 텍스트로 가볍게 변환
- 별도 글래스 칩 없이 순수 텍스트 그리드
- 정보 밀도 높고 깔끔

**장점**: 정보 밀도, 숫자 비교 쉬움, Lovable과 구조 유사하되 경량
**단점**: 모든 항목 없을 때 빈칸 처리

---

#### 옵션 R3: 콤팩트 태그 로우

```
🔥10분  🧊2분  😮‍💨5분  🔄×3
```

- 이모지+값을 태그처럼 가로 나열
- 구분자 없이 이모지가 자연 구분
- R1보다 더 컴팩트 (화살표 없음)

**장점**: 가장 짧고 가벼움
**단점**: 순서/흐름 의미가 약함

---

#### 옵션 R4: 점 구분 원라이너

```
HEAT 10분 · ICE 2분 · PAUSE 5분 · 3세트
```

- 이모지 없이 영문 라벨 + 미드닷(·) 구분
- 퀵로그 폼의 ROUTINE 섹션 영문 라벨과 직접 대응
- 앱 내 다른 곳의 `·` 구분 패턴과 일관 (달력 요약: "3월 · 8회 방문")

**장점**: 폼과 1:1 대응, 앱 내 패턴 일관성
**단점**: 이모지 없어 시각적 구분 약함

---

## 7. 타입별 정보 섹션 분리 — 스토리 카드 참고

### 현재 문제
Quick Log 카드 안에 온도 + 루틴 + 점수가 평면적으로 나열됨.
스토리 카드처럼 **정보의 성격별로 구역을 나누면** 가독성 향상.

### 제안: 3구역 시스템

```
구역 A: 메트릭 (측정값) — 온도, 수치
구역 B: 루틴 (행동)    — HEAT/ICE/PAUSE/RPT
구역 C: 평가 (주관)    — 토토노이, 재방문
```

이 구역은 스토리 카드의 정보 순서와 정확히 대응:
- 스토리: 메인 숫자 → 루틴 뱃지 → (그래프는 상세에 없음)
- 상세: 메트릭 대시보드 → 루틴 → 점수 평가

### 구역 간 분리 방법
- **별도 카드로 분리하지 않음** (카드가 너무 많아짐)
- **하나의 glass-card-light 안에서 시각적 구역 분리**:
  - 구역 A: 대형 숫자 (상단, 패딩 넉넉)
  - 구역 B: 루틴 (얇은 `border-t` 1개로 구분)
  - 구역 C: 점수 (얇은 `border-t` 1개로 구분)

```
┌──────────────────────────────┐
│                              │
│    85°C         15°C         │ ← 구역 A: 메트릭
│    HEAT          ICE         │
│                              │
│ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─  │ ← border-t (얇은 점선 or 실선)
│                              │
│  HEAT  ICE  PAUSE  SET       │ ← 구역 B: 루틴
│  10분  2분   5분    3        │
│                              │
│ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─  │
│                              │
│  토토노이 강도       좋음    │ ← 구역 C: 평가
│  재방문 의향    근처면 갈만   │
│                              │
└──────────────────────────────┘
```

이렇게 하면:
- 스토리 카드의 시각 흐름과 동일 (측정 → 행동 → 평가)
- 퀵로그 폼의 섹션 순서와도 동일 (슬라이더 → 루틴 → 또갈래요)
- 카드 1개로 통합되어 깔끔
